Security & Standards Policy

Overview


Through this Security & Standards (S&S) Policy we seek to build upon existing standards and develop blueprints for the infrastructure of a modern, secure and trust worthy internet based organization.

By setting these standards, we ensure that we maintain consistency and integrity in all our technological operations.

Purpose


The purpose of this document is to define a standard in systems secutiry at Matrix Network International B.V. and build strong security practices into our daily operations.

This includes policies for the following:


- Internal/External Communications

- System Administration & Audits

- Data Protection


Scope


This document lays down a very specific set of requirements for all servers hosted by Matrix Network International B.V.

It also sets out standards in the tooling used by the organization.

Methods of writing secure code, scripts and other hacks are out of the scope of this document.


Requirements


These requirements formulate an audit checklist that a secure system must pass.


System Administration


  • All login attempts are reported to admin.
  • All commands run on the server are reported to admin via a script that must be manually turned on/off.
  • All program error logs are immediately reported to admin.
  • Password and root login are disabled on all systems.
  • SSH access is configured to a non sudo user with least privelage.
  • As a contingency, there exists a means of retrieving critical data, turning down a system and spawning a new instance elsewhere via a single command.
  • Spikes in RAM and/or CPU are immediately reported to admin.
  • File permissions follow the rule of least privelage.
  • Logging follows the Rule of Silence.
  • Production data is never stored on non-production systems.
  • All programs conduct strict sanitization on all inputs from external sources.
  • aide is configured at system genesis.
  • Systems are configured with SELinux & PAM
  • *Careful with that axe, Eugene: rm -ri is always preffered to rm -rf.

Cron


  • aide integrity checks are conducted daily.
  • clamav scans are conducted daily.
  • Packages are updated daily.
  • Log files are compressed daily and filed with a timestamped name.
  • Physical backups of all databases and keys are taken every 28 days.
  • All program logs are audited every 28 days and unrequired data deleted.

Network


  • All programs communicating over the network implements SSL/TLS (either CA approved or Self-signed)
  • Allow incoming connections via port 22, 80 and 443 and deny all other connections.
  • Allow outgoing connections via port 443 and deny all other connections.
  • Incoming connections at port 80 are redirected to port 443.
  • Communications with applications running on other ports are proxied via 443.
  • Port 22 for SSH, only allows connections from whitelisted IPs*
  • IP Forwarding is disabled unless required.

Databases


  • Databases implement users with strict access management.
  • Where possible types are enforced at the database level.
  • Databases use non-standard ports.

General


  • All passwords set across departments must adhere to the password guidlines.
  • Where available, 2FA/MFA is implemented.
  • Cloud backups are configured with the associated service provider; to be as regular as possible.

Tools


  • UNIX Timestamp is the standard organization wide time-unit.
  • SHA256 is the standard organization wide hashing algorithm for signatures and integrity checks.
  • AES256 is the standard organization wide encryption algorithm for files.
  • PGP is used for all email communications.
  • Wire/Telegram/Signal are the preferred E2EE Messengers for private communication
  • Debian/CentOS are the preferred operating systems


International Standards


  • Payment Card Industry Data Security Standard (PCIDSS)
  • General Data Protection Regulation (GDPR)

Password Guidelines


  • Organization wide passwords are generated using a privately discussed format over Wire/Telegram.
  • Passwords must never be shared via mediums that do not implement strong E2EE.
  • Passwords must never be shared in plain text, rather in a privately composed cipher text design.
